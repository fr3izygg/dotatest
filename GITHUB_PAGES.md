# Публикация на GitHub Pages и «общая база»

## Правда ли, что одного localStorage мало для друзей с разных устройств?

**Да, это правда** — если под «общей таблицей» имеется в виду: *каждый открыл сайт у себя на телефоне/ноутбуке и все видят одну и ту же игру в реальном времени*.

| Ситуация | Будет ли общий результат? |
|----------|---------------------------|
| Один компьютер, **несколько вкладок** одного и того же браузера (Chrome и т.д.) на **одном и том же** URL | **Да.** Состояние пишется в `localStorage` этого браузера и дублируется через `BroadcastChannel` между вкладками. |
| Разные люди, **разные устройства** (телефон друга + твой ПК), все открыли `https://…github.io/…/` | **Нет.** У каждого браузера **свой** `localStorage**. GitHub только отдаёт одинаковые **файлы** сайта; он не хранит вашу игру между пользователями. |
| Все смотрят **один экран** (телек/проектор), играет один ведущий с админкой | **Да**, с точки зрения «одна копия состояния» — она на том ПК, где открыта вкладка админа/игроков. |

Итого: **загрузить проект на GitHub и дать ссылку друзьям — сайт откроется и будет работать у каждого**, но **как отдельные одиночные копии**, а не как одна общая онлайн-игра между разными машинами. Чтобы друзья с разных устройств видели **одну** общую игру, нужен **сервер или BaaS** (Supabase, Firebase, свой API + WebSocket и т.д.) — это уже другой объём работы.

## Адрес сайта

Обычный репозиторий `ИМЯ_РЕПО` у пользователя `ВАШ_НИК`:

`https://ВАШ_НИК.github.io/ИМЯ_РЕПО/`

Исключение: репозиторий специального вида `ВАШ_НИК.github.io` публикуется в **корень** `https://ВАШ_НИК.github.io/` — тогда в сборке нужен `VITE_BASE=/` (в workflow ниже для такого случая придётся поправить `VITE_BASE` вручную).

---

## Что уже есть в проекте

- **`vite.config.ts`**: `base: process.env.VITE_BASE ?? "/"` — для GitHub Pages в подпапке задаётся `VITE_BASE=/имя-репо/`.
- **`.github/workflows/deploy-pages.yml`**: при push в ветку **`main`** или **`master`** собирает проект и выкладывает **`dist/`** на GitHub Pages (с правильным `VITE_BASE` из имени репозитория).

## Пошагово: первый раз залить проект на GitHub

### 1. Создать репозиторий

1. Зайдите на [github.com/new](https://github.com/new).
2. **Repository name**: например `dota-quiz` (запомните — это будет часть URL).
3. **Public** (для бесплатного Pages проще всего).
4. Создайте репозиторий **без** README, если будете пушить уже готовую папку с компьютера (или с README — тогда первый push с rebase).

### 2. Залить файлы с компьютера (Git)

В терминале **в папке проекта** (где лежит `package.json`):

```bash
git init
git add .
git commit -m "Initial quiz app"
git branch -M main
git remote add origin https://github.com/ВАШ_НИК/ИМЯ_РЕПО.git
git push -u origin main
```

Подставьте свой ник и имя репозитория. Если GitHub просит логин — используйте **Personal Access Token** вместо пароля ([документация](https://docs.github.com/en/authentication)).

### 3. Включить GitHub Pages на Actions

1. Репозиторий на GitHub → **Settings** → слева **Pages**.
2. **Build and deployment** → **Source**: выберите **GitHub Actions** (не «Deploy from a branch»).
3. Убедитесь, что в репозитории есть файл **`.github/workflows/deploy-pages.yml`** (он уже в этом проекте).

### 4. Запустить деплой

- Сделайте любой **push** в `main` или `master` (или **Actions** → workflow **Deploy to GitHub Pages** → **Run workflow**).
- Вкладка **Actions**: дождитесь зелёной галочки у последнего запуска.
- Через 1–3 минуты сайт: **`https://ВАШ_НИК.github.io/ИМЯ_РЕПО/`** (точный URL также бывает в summary/job **deploy** или в **Settings → Pages**).

### 5. Если сборка упала

- Откройте failed job → шаг **Build** → лог.
- Частые причины: нет `package-lock.json` (у нас он есть — используется `npm ci`), ошибка TypeScript/Vite.

## Сборка у себя на ПК (без Actions)

```bash
npm install
npm run build
```

Для пути `github.io/ИМЯ_РЕПО/`:

**PowerShell:**

```powershell
$env:VITE_BASE="/ИМЯ_РЕПО/"; npm run build
```

Дальше содержимое папки **`dist/`** можно вручную залить в источник Pages (ветка `gh-pages` и т.д.), если не используете Actions.

---

## Локальная разработка

```bash
npm install
npm run dev
```

Браузер: `http://localhost:5173` — **`VITE_BASE` не задавайте** (останется корень `/`).

## Кратко про «сервер и БД на GitHub»

- **GitHub Pages** — это **хостинг статики** (HTML/JS/CSS). Это не Node-сервер и не PostgreSQL.
- **«Как у друга animekvest2»** — это тот же принцип: открывается статический сайт. Если у друга «всё работало с разных телефонов», у того проекта, скорее всего, был **отдельный бэкенд** или другой способ синхронизации, не один только `localStorage` в браузере.

Если позже понадобится **настоящая общая игра по сети**, пишите — можно спроектировать минимальный бэкенд или подключить готовый BaaS под ваш сценарий.
